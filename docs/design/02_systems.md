## 1) 输入采集系统
- 目标：稳定统计键鼠活跃度，作为核心进度来源。
- 采集项（V1）：
- `key_down_count`
- `mouse_click_count`
- `mouse_scroll_step`
- `mouse_move_distance_px`（仅累积距离）
- 关键规则：
- 不记录键值文本、不记录窗口标题。
- 仅统计计数与强度，按 1 秒时间片聚合。
- 超高频输入触发衰减，降低脚本刷分收益。
- 平台策略：
- Windows 优先，使用系统级 Hook（Godot C# 调用 Win32）。
- 非 Windows 平台在 V1 可降级为“仅应用内输入计数”。

## 2) 活动点与资源转换
- 中间资源：`activity_point`（AP）。
- 产出资源：`lingqi`、`insight`、`pet_affinity`。
- 设计原则：
- 转换公式公开可见。
- 存在分钟产出软上限，防止极端输入破坏节奏。
- 允许短时技能提高转化率，但不超过上限阈值。

## 3) 境界与突破
- 字段：
- `realm_level` (int)
- `realm_exp` (double)
- 规则：
- 突破为纯经验驱动：`realm_exp` 满后即可手动突破，无额外材料/悟性消耗，无失败概率。
- 悟性（`insight`）不再参与境界突破，已完全转为子系统熟练度解锁货币（见 §12 子系统熟练度）。
- `breakthrough_pill`、失败保底、成功率提升仍属于 V2 预留设计。
<!-- INSIGHT-MASTERY-CHANGE: 境界突破改为纯经验驱动，悟性完全让给子系统熟练度 -->
- **突破丹处理方案**：
  - V1 决策：取消 `novice_breakthrough_pill`，从掉落表中移除。腾出的掉落 slot 改为灵石/灵草，强化资源循环。
  - V2 预留：若 V2 引入突破失败概率或多阶段突破，突破丹可作为"保底道具"重新上线。

## 4) 探索 / 遭遇 / 战斗规则
- 规则层已拆分为：`BattleStartRules`、`BattleRules`、`BattleLifecycleRules`、`RewardRules`。
- 当前主流程口径：
- 怪物在探索轨道上接近主角。
- 进入触发线后进入战斗态。
- 玩家继续通过输入数推进战斗回合。
- 胜负、奖励、Boss 解锁与战斗日志均通过规则层决策后再由控制器应用。

## 5) 装备、背包与最小奖励闭环
- 已具备 `BackpackState`、`EquippedItemsState`、`EquipmentStatProfile` 的最小闭环。
- 当前装备来源：空存档 starter loadout + 首通固定装备奖励。
- 关键产品边界：
- 新装备进入背包，不自动替换已装备物品。
- 仅在玩家手动触发装备动作时才替换同槽位旧装备。
- 当前仍属原型/开发验证：未实现完整装备内容池、随机词条、强化和最终换装 UX。

## 6) 双窗口桌面 UI 系统
- 主横向窗口：
- 常驻桌面底部，承载探索与战斗进度演出。
- 左上按钮用于拖动窗口位置。
- 右上按钮用于调整主窗口横向宽度。
- 左下图书按钮用于打开/关闭子菜单窗口。
- 图书按钮下方展示当前修炼阶段。
- 右下展示当前区域探索进度条。
- 子菜单窗口：
- 独立窗口，采用书页/卷轴形态，顶部为左右两组页签。
- 左侧页签组（游戏内容）：`修炼概况`、`战斗日志`、`装备情况`、`背包`、`统计概览`、`配置校验`。
- 右侧页签组（功能设置）：`Bug反馈`、`设置`。
- 页签内容区采用单页全宽展示（不再左右分栏同时显示）。
- 左上角提供关闭按钮（`X`）。
- 底部状态条固定显示 `灵石数量`。
- 可单独开关，不中断主窗口探索循环。
- `战斗日志` 承载最近战斗记录，不再额外占用主横向窗口常驻区域。
- 战斗日志默认保留最近 10 条，按时间倒序展示，支持显示怪物、结果、掉落与资源结算摘要。
- Demo 阶段隐藏 `联机`、`管理员模式`、`手写支持`、`开机启动动画` 等未形成闭环的入口，避免把预留项暴露为可用功能。
- `Bug反馈` 提供最小闭环：问题描述输入、复制日志路径、导出反馈文件、打开数据目录。
- `配置校验` 页签展示 `LevelConfigLoader` 的校验结果，支持按范围与当前关卡过滤查看，并可直接执行 200/1000 次掉落模拟。
- 设置页对未完全接线的选项使用 `（预留）` / `（实验）` 标识，降低误解成本。
- 可选 `修行节律提醒`：按 25/45/60/90 分钟等周期对当前活跃会话做轻量提醒，将“番茄钟式休息提示”与本轮修炼进度摘要合并展示。
- 交互约束：
- 默认低干扰，面板状态与窗口尺寸需持久化。
- 窗口缩放后核心信息（修炼阶段、探索进度）必须可见。
- UI 文案统一从 `scripts/ui/UiText.cs` 提供，避免分散硬编码与乱码问题。

### 6.1 修行节律提醒（Cultivation Rhythm Reminder）
- 定位：可选的节律层，而非强制打断层。目标是在不破坏桌宠低打扰定位的前提下，为长时挂机提供“休息建议 + 进度回看节点”。
- 默认策略：默认开启 `弱提醒`，避免首次使用就出现强打扰。
- 提醒强度：
  - `无提醒`：完全不主动提示，仅在玩家主动打开界面时查看累计进度。
  - `弱提醒`：到点后显示非阻断提醒，例如 Toast、底栏高亮、书本未读徽标。
  - `强提醒`：到点后触发更明显的提醒，例如窗口 attention、居中提示或短时高亮；仍不得抢占焦点或强制暂停。
- 周期配置：支持 `25 / 45 / 60 / 90` 分钟预设，并允许后续扩展自定义分钟数。
- 展示内容：
  - 本轮获得 `灵气 / 悟性 / 境界经验`
  - 当前主行为推进摘要（如探索进度、炼丹批次、采集产出）
  - 本轮关键事件计数（突破、精通解锁、装备掉落、Boss 击杀等）
  - 一句休息建议文案，例如“已连续修行 50 分钟，建议起身活动片刻”
- 设计边界：
  - 进度摘要必须服务于“回看这一轮收获”，而不是把产品改造成独立效率工具。
  - 不得使用会抢占输入焦点的系统模态弹窗。
  - 在 `工作模式` 下默认只保留 `弱提醒` 或 `无提醒` 两档，避免干扰主工作区。
- 与通知体系关系：
  - `弱提醒` 归入三层通知体系中的“瞬时提醒 + 未读回看”组合。
  - `强提醒` 仅在用户明确开启时，才叠加窗口 attention 等更高感知手段。
  - 每个提醒周期结束时应向事件日志写入一条“修行周期总结”记录，便于用户回看。

## 7) 灵宠成长预留
- 当前 V1 不将灵宠系统纳入主成长闭环；完整的亲密度等级、互动/羁绊设计已整体延后到 V2。
- `pet_affinity` 在 V1 中不承担等级成长、全局增益或主动培养职责，最多作为轻量累计统计存在。
- `pet_mood` 不要求玩家维护，也不作为主 UI 重点展示项；但当前代码中仍会影响资源转化倍率，其后续定位待 V2 与灵宠互动/羁绊玩法一起重构。
- 若后续开启灵宠互动玩法，再补充心情恢复、衰减、羁绊被动等完整闭环。

## 8) 存档与防刷
- 存档结构：单文件 `user://save_state.cfg`（ConfigFile）+ `meta.version`。
- 必要字段：
- `meta.version`
- `meta.last_saved_unix`
- `ui.main_bar_x`
- `ui.main_bar_width`
- `ui.submenu_visible`
- `ui.submenu_active_left_tab`
- `ui.submenu_active_right_tab`
- `combat.recent_logs`
- `equipment.equipped`
- `inventory.__equipment_profiles`
- `input.stats`
- `input.hook_paused`
- 迁移策略：
- 优先读取统一存档；若不存在则自动迁移旧 `ui_state.cfg` + `game_state.cfg`。
- 防刷策略：
- 检测异常高频峰值并打标。
- 本地时间跳变超阈值时进入保守结算。
- 长时间无焦点但高输入速率时降低收益权重。

备注（当前状态）：上述防刷目标已在设计中明确；当前规则层已落实高频衰减、掉落 soft-cap / daily-cap 等基础护栏，异常时间跳变与长时间无焦点场景仍需要后续专项补齐与验收。

## 9) 炼丹系统（Alchemy / Potion Crafting）
- 设计灵感：参考 Melvor Idle 的 Herblore，将副本掉落的草药材料转化为战斗消耗品。
- 定位：资源消费端 — 消耗探索掉落的 `灵草` 类材料 + `灵气`，产出战斗可用的丹药。
- 活动模式：玩家在底栏切换到"炼丹"模式后，键鼠输入转化为**炼丹进度**而非探索进度。
- 核心机制：
  - 选择配方 → 消耗材料 → 输入驱动进度条 → 完成后产出丹药进入背包。
  - 每批次需要固定输入量（如 200 次键鼠事件 = 1 批丹药）。
  - AP 仍正常结算灵气/悟性，但探索不推进。
- V1 丹药列表：
- `回气丹`：战斗中 HP < 50% 时自动使用，恢复 30% 最大 HP。消耗：`spirit_herb` ×2 + 灵气 ×50。
- `聚灵散`：下次战斗灵气掉落 +25%。消耗：`spirit_herb` ×3 + 灵气 ×80。
- `破境丹`（后续）：突破时经验需求 -10%。消耗：高阶草药 + 灵气 ×200。
- 丹药使用规则：
  - 战斗消耗品在进入战斗时自动检查并消耗，不需手动操作。
  - 背包中同类丹药堆叠显示数量。
- 存档字段：`alchemy.selected_recipe`, `alchemy.progress`, `backpack.potions[]`。
- 离线支持：当前实现中炼丹模式不会离线推进配方进度；离线仍按修炼式 AP 结算处理，若后续需要真实离线炼丹产出需单独扩展。

## 10) 炼器 / 强化系统（Smithing / Equipment Enhancement）
- 设计灵感：参考 Melvor Idle 的 Smithing，将副本掉落的碎片/碎符材料转化为装备强化。
- 定位：资源消费端 — 消耗 `碎片`、`碎符` 类材料 + `灵气`，提升已有装备的 `EnhanceLevel`。
- 活动模式：玩家在底栏切换到"炼器"模式后，键鼠输入转化为**强化进度**。
- 核心机制：
  - 选择目标装备 → 选择强化配方 → 消耗材料 → 输入驱动进度条 → 完成后 `EnhanceLevel + 1`。
  - 强化上限 = `稀有度等级 × 3`（俗器 3 级，法器 6 级，灵器 9 级，宝器 12 级）。
  - 每级强化提升装备基础属性 8%（乘算叠加）。
  - 每批次需要输入量随强化等级递增（100 + EnhanceLevel × 50 次键鼠事件）。
- V1 强化材料需求：
  - +1~+3：碎片 ×3 + 灵气 ×30
  - +4~+6：碎片 ×5 + 碎符 ×1 + 灵气 ×80
  - +7~+9：碎片 ×8 + 碎符 ×3 + 灵气 ×150
- 存档字段：`smithing.target_equipment_id`, `smithing.progress`。
- 离线支持：当前实现中炼器模式不会离线推进强化进度；离线仍按修炼式 AP 结算处理，若后续需要真实离线炼器产出需单独扩展。
- 注意：`EquipmentInstanceData.EnhanceLevel` 字段已存在于代码中但未消费，此系统将其激活。
<!-- SUBSYSTEM-EQUIP-EXPAND: 炼器新增锻造子路线与跨系统强化辅料 -->
- **锻造子路线（新增）**：炼器模式下新增"锻造"路径，消耗矿石/灵花/碎符等材料直接锻造基础装备（俗器/法器），与"强化"路线并行。
  - 锻造配方 10 个，覆盖 3 槽位 × 4 系列（寒铁锻造系/灵蕊滋养系/符文铭刻系）。
  - 俗器配方炼器 Lv1 可用，法器配方需炼器精通 Lv2，符文系需额外符箓精通 Lv2。
  - 详见 `11_equipment_content_system.md` §15.6 与 `12_equipment_sample_qi_refining.json` `forge_recipes[]`。
- **跨系统强化辅料（新增）**：+4 以上强化引入采集系辅料需求：
  - +4~+6 新增寒铁矿×2（矿脉）
  - +7~+9 新增灵玉×1（矿脉）+ 灵花×2（灵田）
  - +10~+12 新增秘银×2（矿脉）+ 灵果×1（灵田）+ 灵珠×1（灵渔）
  - 辅料数量 < 主材数量，不压过碎片/碎符的主体地位。

## 11) 灵石经济系统（Spirit Stone Economy）
- **问题背景**：灵石显示与产出链路现已接入，消费端已扩充至覆盖 12 系统。
- 定位：灵石是**通用中间货币**，连接掉落端和消费端。
- 产出来源：
  - 战斗胜利基础灵石奖励（每场 5-15 灵石，按区域缩放）。
  - 出售多余材料（设计预留，当前未形成完整 UI/交互闭环）。
  - 首通奖励（60-200 灵石，按区域递增）。
  - Boss 首杀奖励（300-500 灵石）。
- 消费去向（按系统分类）：
  - **装备兑换**：区域材料兑换配方（购买指定部位基础装备，50-200 灵石）。
  - **Boss 重挑**：非首次可用灵石跳过冷却（50 灵石/次）。**V1 后续**。
  - **炼丹/炼器加速**：灵石替代部分材料缺口（保守定价防止 P2W 感）。**V2 预留**。
  - **丹药急购**：背包无回气丹时可花 30 灵石直购 1 个。**V1 后续**。
  <!-- SUBSYSTEM-EXPAND: 新增消费端 -->
  - **种子商店**（灵田）：购买高级种子（普通种子 20 灵石，稀有种子 80 灵石，史诗种子 200 灵石）。**V1**。
  - **矿脉刷新**（矿脉）：重置当前矿脉矿点分布（30 灵石/次），有概率刷出稀有矿脉。**V1**。
  - **灵塘扩容**（灵渔）：解锁额外钓点位（第 2 钓点 100 灵石，第 3 钓点 300 灵石）。**V1 后续**。
  - **符咒急购**：背包无指定符咒时可花 50 灵石直购 1 张基础符咒。**V1 后续**。
  - **灵膳急购**：背包无灵膳时可花 40 灵石直购 1 份基础灵膳。**V1 后续**。
  - **阵盘快速切换**：战斗中切换阵盘收取 20 灵石手续费（非战斗切换免费）。**V1**。
  - **悟道冥想加速**：消耗 100 灵石强制触发一次顿悟事件。**V2 预留**。
  - **体修恢复药浴**：消耗 60 灵石跳过锻体冷却时间。**V1 后续**。
- 出售材料约束：
- 若后续补上出售操作，需弹窗确认并显示出售数量与获得灵石数。
- 当前代码尚无 `BackpackState.SellItem()` 与对应确认弹窗实现。
- 存档字段：`wallet.spirit_stones`。
- 代码影响：
  - `ResourceWalletState` 新增 `SpiritStones` (int) 字段及 `AddSpiritStones()` / `SpendSpiritStones()` 方法。
  - `RewardRules.BuildBattleRewardSummary()` 新增灵石显示。
  - 当前版本仍缺少稳定的灵石消费调用点，后续需补齐出售、兑换或快捷消费入口。

循环说明：
- **采集系**（副本/灵田/矿脉/灵渔）产出 16 种原材料，是资源蛛网的源头。
- **加工系**（炼丹/炼器/符箓/烹饪/阵法）消耗采集材料 + 灵气，产出战斗消耗品/增益/装备强化。
- **修行系**（修炼/悟道/体修）提升基础能力：修炼=灵气+经验，悟道=永久属性，体修=永久HP/DEF。
- **灵气**：所有模式被动恢复 5/min + 修炼高效产出，是加工系统的通用燃料。
- **悟性**：所有模式按 AP 产出，用于 12 系统熟练度解锁（见 §12）。
- **材料蛛网**：每种材料被 1-3 个系统消费，每个加工系统从 2-4 个来源取材料 → Melvor 式互联。
- 12 模式闭环：**采集(材料) → 加工(消耗材料→成品) → 战斗(消耗成品) → 推进更高副本(更多材料)**。

## 12) 子系统熟练度系统（Subsystem Mastery）
- 设计灵感：参考 Melvor Idle 的技能等级体系 — 每个活动系统有独立等级，等级解锁更高级的行动和被动加成。
- 核心概念：悟性（Insight）作为唯一解锁货币，永久解锁各子系统的熟练度等级。
- 定位：悟性不再用于境界突破，完全服务于子系统横向成长。

### 12.1 设计原则
- **一次性解锁**：花费悟性 → 永久获得该等级效果，无需维护。
- **低打扰**：解锁后自动生效，不需要额外操作。
- **有意义的选择**：悟性有限，玩家需要决定优先提升哪个系统。
- **与境界联动**：部分高级熟练度需要境界等级达标才可解锁。

### 12.2 熟练度定义

#### 副本精通（Dungeon Mastery）

|------|---------|---------|------|

| Lv3 | 60 | 炼气三层 | 寻宝直觉：材料掉率 +15% |

| Lv3 | 50 | 炼气三层 | 双修并行：修炼时缓慢推进探索进度（0.3 倍速） |

### 12.3 数值总览
- 原 4 系统 Lv2-Lv4 解锁合计：30+60+100 + 25+50+80 + 20+45+75 + 25+50+80 = **640 悟性**
- 新 8 系统 Lv2-Lv4 解锁合计：~**1440 悟性**（各系统见 §13-§20）
- 全部 12 系统解满总计：640 + 1440 = **~2080 悟性**
- 轻度用户日产悟性：~48
- 全解锁时间：~2080 / 48 ≈ **43 天**（~6 周）
- 实际节奏（优先解锁当前瓶颈）：前 2 周解锁 8-10 个低级熟练度，后 4 周逐步解锁中高级
- 设计目标：玩家每 1-3 天解锁一个新能力，保持"每隔几天发现有新东西"的新鲜感。
- 详细悟性消费表见 `03_progression_and_balance.md`「悟性消费设计」。

## 13) 灵田系统（Spirit Garden）
- 设计灵感：参考 Melvor Idle 的 Farming — 种植→等待→收获循环，最适合离线收益。
- 定位：采集端 — 稳定产出灵草/灵花/灵果/种子，减轻对副本灵草的依赖。
- 活动模式：玩家切换到"灵田"模式后，键鼠输入驱动种植和浇灌进度。
- 核心机制：
  - 选择作物 → 消耗种子+灵气 → 输入驱动生长进度 → 成熟后收获材料。
  - 每种作物有不同生长周期（输入需求）和产出。
  - 收获时有小概率获得稀有种子（递归循环）。
- V1 作物列表：
  - `灵草种` → 灵草 ×2-3，生长 150 次输入
  - `灵花种` → 灵花 ×1-2，生长 200 次输入
  - `灵果种` → 灵果 ×1，生长 300 次输入
- 种子消耗出口（防止无限囤积）：
  - **种回田中**：常规循环消耗（每次种植消耗 1 种子）。
  - **炼丹辅料**：高阶丹药配方可选消耗种子提升品质（如：灵草种 ×2 + 灵草 ×3 → 高品质回气丹，HP 恢复 +10%）。
  - **育种研究**：消耗 5 颗同类种子 → 永久提升该作物收获量 +5%（上限 +50%，计 10 次）。存档字段：`garden.breeding_level{}`。
  - **灵石出售**：多余种子可在种子商店回售（普通种子 5 灵石，稀有种子 20 灵石）。
- 离线支持：灵田模式下离线可继续生长（按离线时长折算进度，效率 0.6 倍）。

## 通用活动系统框架（Extensible Activity Framework）

### 设计目标
所有采集/加工/修行活动共享同一套运行框架，新增子系统只需定义**配方数据**即可接入，无需修改框架代码。

### 框架接口
```
// 每个活动系统实现此接口
interface IActivityDefinition {
    string SystemId;           // 唯一标识 "alchemy", "mining", etc.
    string DisplayName;        // 中文展示名
    ActivityCategory Category; // Gathering / Processing / Cultivation
    bool SupportsOffline;      // 是否支持离线推进
    IReadOnlyList<IRecipeDefinition> Recipes; // 该系统的所有配方/任务
}

interface IRecipeDefinition {
    string RecipeId;
    string DisplayName;
    int RequiredInputEvents;         // 完成一批需要的键鼠事件数
    IReadOnlyList<MaterialCost> Costs; // 材料消耗列表
    double LingqiCost;               // 灵气消耗
    IReadOnlyList<MaterialYield> Yields; // 产出列表
    int RequiredMasteryLevel;        // 需要的精通等级(1=无门槛)
}

enum ActivityCategory { Gathering, Processing, Cultivation }
```

### 运行框架
```
CraftingProgressionService (已有，扩展为通用版):
  - RegisterActivity(IActivityDefinition def)
  - AdvanceProgress(string systemId, int inputEvents)
  - TryCompleteRecipe(string systemId) → yields or null
  - GetProgress(string systemId) → (current, required)
```

### 扩展步骤（新增一个子系统）
1. 新建 `XxxDefinition.cs`：实现 `IActivityDefinition`，定义配方列表
2. 新建 `XxxState.cs`（若需持久化特殊状态）或复用通用 `ActivityProgressState`
3. 在 `ActivityRegistry` 中注册
4. 在 `PlayerActionState` 中新增模式常量
5. 存档迁移：补齐新字段
6. UI：在模式选择器中增加一项

**预估代码量**：每个新系统约 80-150 行定义代码 + 1 个测试文件。

### 通用活动进度存档
所有活动使用统一的存档结构：
```
activity.{systemId}.selected_recipe  (string)
activity.{systemId}.progress         (double)
activity.{systemId}.custom.*         (系统特有字段)
```

### 材料注册表
所有材料在统一的 `MaterialRegistry` 中注册，每种材料定义：
- `material_id`（唯一标识）
- `display_name`（中文名）
- `stack_limit`（堆叠上限）
- `source_systems[]`（产出来源系统列表）
- `consumer_systems[]`（消耗去向系统列表）
- `sell_value`（出售灵石价值，0 = 不可出售）
